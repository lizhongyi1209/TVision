## 环境与工具链（Windows / 编码 / 构建）

### [2026-07-08] 中文 .bat 双击报「不是内部或外部命令」
- 现象：生成的批处理逐行被 cmd 切碎，中文乱码。
- 根因：cmd 解析 .bat 要求 **GBK + CRLF**；Write 工具默认落盘 UTF-8 + LF。
- 解决：写完后用 iconv/PowerShell 转 GBK+CRLF；`启动 TVision.bat` 已是转换后版本。
- 预防：任何 .bat 新建/修改后必须转码；勿用会改编码的编辑器另存。

### [2026-07-09] next build 打挂了正在跑的 next start
- 现象：验证性 `npm run build` 几分钟后，用户同目录 `npm run start`（:3000）进程消失。
- 根因：Windows 文件锁比 POSIX 严格，`next build` 覆写 `.next` 与 `next start` 持有的句柄冲突，后者非正常退出。
- 解决：用户重启启动器恢复。**2026-07-12 复发**：用户此前答复过「服务无所谓，直接 build」，当天第二次 build 后 `next start`（:3000）进程真的消失了——当场 `npm run start` 后台重新拉起（顺带跑上了新构建产物），netstat + curl 200 验证恢复。
- 预防：明知有人正用同目录 `next start` 时**不要**重复 build；要验证就换独立输出目录或等服务停了再 build。若用户已表态服务可牺牲而 build 后进程消失，**build 完立即重启服务并验证**，别留一个挂掉的 3000 端口给用户。

### [2026-07-09] GitHub push 直连被重置
- 现象：`git push` 连接 github.com 被 reset。
- 根因：本机直连不通，需走本地代理。
- 解决：先问用户**当前**代理端口，用 `git -c http.proxy=... push` 一次性传参。
- 预防：勿把代理写进 git 全局/仓库配置（端口会变）；每次 push 前现问现用。

### [2026-07-09] Phosphor 图标存在性探测假阴性
- 现象：`node -e "require('@phosphor-icons/react')..."` 探测 `PaintBrush` 报不存在，但实际可用。
- 根因：该包 CJS 入口与 ESM 导出不一致；webpack/Next 实际走 ESM 解析。
- 解决：用 `node --input-type=module` 动态 `import()` 探测，以其结果为准。
- 预防：探测手段必须与构建工具的模块解析方式一致。

## 前端 / React

### [2026-07-08] react-image-crop 图片高度类失效
- 现象：img 上的 max-h 类不生效，裁剪图溢出。
- 根因：库自带 CSS 用 `max-height: inherit` 顶掉 img 的高度类。
- 解决/预防：高度上限挂在 `ReactCrop` 根节点上，不挂 img。

### [2026-07-09] toast 连续两条只显示后一条
- 现象：同步连续调用 `showToast` 前一条被顶掉。
- 根因：toast 是单槽位状态。
- 解决/预防：需要连续提示时用 `setTimeout` 错开（现有约定 3s），或合并成一条。

### [2026-07-09] 受控输入的值与提交时闭包不一致（疑难）
- 现象：备注输入后点确认，读到的是旧值（未能稳定复现，疑输入法组字时序）。
- 根因：极端时序下 React 状态与 DOM 现值可能不同步；事后在函数内 `.blur()` 救不回已捕获的闭包值。
- 解决：提交时按 `data-*` 定位输入框读 **DOM 真实值**与 state 比对，不一致以 DOM 为准并 diag warn 留痕。
- 预防：关键提交路径对用户输入做"眼见为实"核对；这个模式可复用。

### [2026-07-09] 选中即卸载的组件里放异步编排 → 无法取消
- 现象：RadialMenu 点击后立刻卸载，视觉反推请求没法真正 abort。
- 根因：发起方组件生命周期太短。
- 解决：编排搬到常驻父组件（GenerateBar）的 effect，`AbortController` + `cancelled` 闭包双保险，依赖键含 `visionRequestId` 支持重试重触发。
- 预防：异步编排放在生命周期覆盖整个请求的组件里。

### [2026-07-12] 嵌套 onDrop 未挡冒泡，内层拖放会连带触发外层重置
- 现象：（实施多参考图时代码走查发现，未等真实复现就已修复）参考图栏的上传 / 追加入口嵌套在 Stage 画布外层的 `onDrop` 容器里；若直接拖文件到内层框，内层与外层的 `onDrop` 会同时触发——外层把这批文件当"新主图 + 新参考图"整体重置一遍，刚加进内层的参考图又被清空、主图也被换掉。
- 根因：原生 drop 事件默认冒泡，React 合成事件同样按冒泡顺序依次调用各层 `onDrop`，除非某一层显式 `stopPropagation()`；两层都只调了 `preventDefault()`，没调 `stopPropagation()`。这个缺口在预设模式的单参考大上传框里本就存在，只是触达面很窄（只有选中 needsRef 操作才露出），自由模式把"＋ 添加参考图"变成常驻入口后才会被高频触发。
- 解决：`RefSlot.tsx` 里两个真正承载"追加 / 局部替换"语义的入口（`PresetRefBox`、`FreeRefList` 各自的上传 / 添加框）在各自 `onDrop` 里补 `e.stopPropagation()`；参考图缩略图本身不拦截——按设计，直接拖到缩略图上冒泡触发外层整体重置是预期行为（D5：舞台拖入 N 张 = 重置），只有"更具体的追加/替换操作"入口才需要拦。
- 预防：一个可拖放区域嵌套在另一个语义不同的可拖放区域内部时（内层"追加/局部替换" vs 外层"整体重置"），内层必须 `stopPropagation()`，否则两个处理器都会跑；新增任何嵌套拖放入口前先检查这一点。

### [2026-07-12] 参考图列表缩略图 object-cover 裁掉内容
- 现象：自由模式参考图列表里，竖版人像参考图被横向小框裁得只剩局部（用户截图反馈：只剩肩膀或下巴一小条，看不出参考图原貌）。
- 根因：`RefSlot.tsx` 的 `FreeRefList` 缩略图用 `object-cover` 配**固定高度**的横向框，宽高比不匹配时裁切填满会丢内容；同文件里预设模式的大预览框（`PresetRefBox`）本来就用 `object-contain` + 只设上限高度（不强制外框），自由模式的列表项新写的时候没照抄这一点。
- 解决（分两步，第二步才是最终方案）：① 先把 `object-cover` 换成 `object-contain` 止血——图片不再被裁，但外框仍是固定高度的宽矩形，人像图被挤成中间一条窄图、两侧一大截黑边，用户反馈"外框总是长方形，不好看"；② 用 `design-taste-frontend`（taste-skills）里的 Shape Consistency 思路（同一文件内已有的展示逻辑要复用，不要各写一套）重新设计：**去掉固定高度**，缩略图 `w-full` + `object-contain` + 只设 `max-h`（数值直接复用 `PresetRefBox` 同一文件里已有的 `compact ? "max-h-[200px]" : "max-h-[52vh]"`），让卡片贴合每张图片自己的真实比例——竖图卡片自然窄高、横图卡片自然宽矩，`max-h` 只在极端比例时才兜底触发 `object-contain` 的留白，常规参考图零留白。方案选型经 `AskUserQuestion` 给用户出了 3 个 ASCII 草图（自适应比例 / 双栏方形网格 / 单列 3:4 定框）确认后落地。
- 预防：① 图片展示优先级 = 贴合真实比例（无外框）> 按内容常见比例设外框 > 固定通用外框；只有明确要统一网格感才反过来。② 遇到"图片被挤/留大片黑边"类反馈，根因通常是"外框比例强制 ≠ 内容比例"，先看同文件/同项目里有没有已验证的"贴合比例"展示模式可以直接复用，而不是急着调 `object-fit` 参数了事。③ 涉及纯展示/观感取舍的改动，落地前先出 ASCII 草图让用户选，比直接改代码再来回改省时间。

### [2026-07-12] 参考图列表用滚动容器兜多图，用户预期是"全部一屏可见"
- 现象：自由模式参考图列表（`max-h-[54vh] + overflow-y-auto` 单列竖排）加到四五张后，「＋ 添加参考图」入口被顶出可视区，必须滚到底才能继续加图；用户明确反馈操作不便，预期"9 张（含主图）全部在画布中展示，不要滚动条"。
- 根因：设计时按"列表长了就滚"的默认思路给了滚动兜底，没按上限张数（当时 6，现调至 8）反推布局是否放得下——滚动容器藏住了溢出，也把常用入口（追加按钮）一起藏了。
- 解决：`FreeRefList` 重做为**自适应双列 · 无滚动**（用户 3 选 1 草图确认）：≤3 张单列、≥4 张自动切双列；每张缩略图高度上限 = `min(52vh|200px, (100vh - 380px) / 行数)` —— 380px 是舞台行上下留白（32+232）+ 添加条 + 行间隙的总预算，行数 = `ceil(张数/列数)`，故 8 张 + 添加入口任何窗口高度下都一屏放下；卡片仍贴合图片真实比例（外层 grid 定车道、内层 `w-fit max-w-full` 收缩包裹图片，延续上一条的自适应结论）；双列时移序按钮换 `CaretLeft/CaretRight`（视觉流向变了，上/下箭头会误导），compact 态只留更换/删除（车道约 71px 宽放不下 4 个按钮，规格对齐 `PresetRefBox` compact）。同轮 `MAX_REF_IMAGES` 6 → 8。
- 预防：① 有明确数量上限的列表，先按**上限**画满员布局验证放得下，再决定要不要滚动——滚动是最后手段，不是默认兜底；尤其不能让"追加"这类高频入口随内容滚出屏。② 固定尺寸改按数量均分时，预算公式（视口 - 固定占位）写进注释，别留魔法数。③ 每图张数上调会放大"上游模型多图上限未实测"的风险（见下方待实测条目），改上限时同步提醒用户。

## 图像 / Canvas

### [2026-07-09] destination-in 羽化完全失效，回贴成硬边矩形
- 现象：局部重绘回贴边缘生硬。
- 根因：`globalCompositeOperation="destination-in"` 吃的是 **alpha 通道**，不是 RGB 亮度；黑底白线的"黑白遮罩"全图 alpha=255，等于没遮。
- 解决：mask 画布改**透明背景 + 不透明白描边**再整体模糊，让 alpha 本身承载羽化。
- 预防：canvas 合成运算先查清楚吃 alpha 还是吃颜色。

### [2026-07-09] canvas 自模糊自绘有花屏风险
- 现象：对同一画布"blur 后画回自己"跨浏览器行为未定义。
- 解决/预防：`ctx.filter = blur()` 绘制到**第二块离屏 canvas**，结果等价且稳定。

## API / 上游

### [2026-07-09] 视觉接口 400：reasoning_effort 与 thinking_level 混淆
- 现象：带 `thinking_level` 调 `/v1/chat/completions` 被 400。
- 根因：同一网关两套端点参数不同——`/async/v1/generateImage` 用 `thinking_level`，OpenAI 兼容聊天端点用 `reasoning_effort`。
- 解决：`reverseEngineerPrompt()` 遇 400 自动去参对同模型重试一次，再失败才换模型。
- 预防：多端点网关的请求字段逐端点核对官方文档，不要互相套用。
