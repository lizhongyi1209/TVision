# 任务模式设计方案

状态：方案已确认，当前开发快照已备案；详见 `PLAN-TASK-MODE-STATUS.md`。

## 1. 产品定位

“任务模式”是一套可保存、可复用、可追踪的图片工作流编辑器。用户定义每一步执行的操作，并把前一步输出绑定到后一步输入，运行时系统按顺序自动完成整条流程。

首版采用自动排列的纵向步骤流，不开放无限画布、任意分支和循环。底层数据仍按有向无环图设计，为后续条件分支保留扩展空间。

核心概念：

- 流程：用户保存的可复用定义。
- 步骤：流程中的一个执行节点。
- 运行：流程的一次执行实例，固定使用启动时的流程快照。
- 变量：任务输入或前序步骤输出，分为 `text`、`image`、`image[]`。

## 2. 典型流程

“反推 + 自定义词 + 新参考图生成”：

1. 任务输入：原始图、参考图、自定义描述。
2. 视觉反推：读取原始图，输出反推提示词。
3. 提示词组合：合并反推提示词、自定义描述和固定文本。
4. 图片生成：读取最终提示词和参考图，执行生成。
5. 任务输出：选择生成结果，并同步到历史生成。

## 3. 界面结构

任务模式作为顶栏一级工作区，与单图创作、批量工坊并列。

- 左栏：我的流程、流程模板、最近运行、节点库。
- 中间：自动排列的步骤流，支持拖动排序和在步骤之间插入。
- 右栏：所选步骤的输入映射、参数、输出预览和运行策略。
- 底部：当前运行进度、当前步骤、停止后续步骤、查看详情。
- 顶部：流程名、实时检查结果、设计/运行记录切换、保存、运行。

输入映射通过下拉选择完成，不要求用户手动画线或记变量语法。系统根据选择自动显示数据关系，并按类型过滤可选来源。

## 4. MVP 节点

### 任务输入

- 用户可定义文本、单图片字段。
- 每个字段支持名称、必填、默认值。
- 运行前只展示该流程需要填写的字段。

### 视觉反推

- 输入：`image`。
- 输出：`text`，同时保留原始结构化 JSON。
- 复用现有视觉反推能力。

### 提示词组合

- 输入：一个或多个 `text`。
- 输出：`text`。
- 编辑器内以变量标签展示来源，不直接暴露易出错的裸变量语法。
- 保存模板文本、变量绑定和最终渲染结果。

### 图片生成

- 输入：提示词、可选底图、可选参考图。
- 输出：`image[]`。
- 复用现有模型、分辨率、比例、计费、质量、张数参数。
- 换上衣、换背景、白底图等现有操作作为该节点的“生成配方”，不拆成大量独立节点。

### 任务输出

- 输入：任意前序输出。
- 对 `image[]` 必须选择具体图片，首版不自动遍历。
- 最终图片继续进入现有历史生成，并增加流程与运行关联。

首版不支持画笔遮罩类步骤。物品移除需要人工选区，不适合直接进入无人值守流程。

## 5. 检查与运行规则

编辑时持续检查，阻断问题包括：

- 必填输入未绑定。
- 文本、图片或图片列表类型不兼容。
- 使用了位于当前步骤之后的输出。
- 模型、分辨率、比例、计费组合非法。
- 参考图超过限制。
- 图片列表未指定具体图片。
- 未配置任务输出。
- 流程存在环或超过步骤上限。

运行规则：

- 首版最多 12 个步骤。
- 同一用户同时只运行一个流程，步骤顺序执行。
- 启动时保存流程快照，之后编辑不影响本次运行。
- 已成功步骤输出可复用，失败重试不重复执行上游付费步骤。
- “停止”准确命名为“停止后续步骤”；已提交给上游的当前任务可能无法真正取消。
- 支持运行到此步骤、重试本步骤并继续、从此步骤重新运行。
- 每个步骤配置超时、最大轮询时间和有限重试次数。

节点状态：等待、运行中、成功、失败、已阻断、已跳过。

## 6. 运行记录

每次运行保存：

- 流程 ID、流程版本和完整快照。
- 任务输入快照。
- 每步参数、输入绑定、开始/结束时间、状态、错误和尝试次数。
- 上游任务 ID、文本输出、图片资产引用。
- 最终输出和总耗时。

运行记录支持只读回放、从失败处继续、使用相同输入再次运行。最终图片仍可下载、设为单图画布和在历史生成中查看。

## 7. 技术设计

任务模式必须使用独立的服务端执行器，不能通过前端模拟点击单图创作页面来完成。

建议模块：

- `taskStore.ts`：编辑态、当前运行摘要和页面状态。
- `workflowTypes.ts`：流程、节点、绑定、运行和资产类型。
- `workflowRegistry.ts`：节点元数据、输入输出端口和参数校验。
- `workflowRunner.server.ts`：拓扑校验、步骤调度、重试、恢复和状态持久化。
- `workflowStore.server.ts`：流程定义和运行记录的原子读写。
- `workflowAssets.server.ts`：本地图片 URL、文件路径和 data URL 的统一转换。

建议 API：

- `GET/POST /api/workflows`
- `GET/PUT/DELETE /api/workflows/:id`
- `POST /api/workflow-runs`
- `GET /api/workflow-runs/:id`
- `POST /api/workflow-runs/:id/stop`
- `POST /api/workflow-runs/:id/retry`

关键实现约束：

- 现有生成结果返回的是本地 `/api/media` URL，下一生成步骤不能把该 URL 直接发给外部模型；服务端必须先解析为本地文件并转换成 data URL。
- 生成提交使用部分成功可记录的策略，不能让一个请求失败掩盖其他已提交且可能已计费的任务。
- 运行状态先持久化再提交上游任务，避免孤儿任务。
- 文件持久化必须使用原子写或串行锁，避免并发覆盖历史元数据。
- 每次执行使用 attempt ID / 幂等键，防止网络重试造成重复付费提交。
- 如果产品未来支持多用户，流程、历史、模板和资产都要按用户隔离。

## 8. 开发拆分

### 第一阶段：流程编辑器

- 新增任务模式入口和三栏布局。
- 完成五类节点、排序、插入、删除、复制。
- 完成类型化输入映射、实时校验、流程 CRUD 和流程模板。

### 第二阶段：执行器

- 抽出现有反推、生成、轮询和图片解析服务。
- 完成服务端运行记录、顺序调度、恢复、停止后续步骤和失败重试。
- 接入运行前输入面板和底部进度条。

### 第三阶段：历史与质量

- 运行回放、相同输入再次运行、从失败处继续。
- 与现有图片历史互通。
- 单元测试、服务端恢复测试、并发与重复提交测试。

## 9. 首版明确不做

- 条件分支、循环、并行步骤。
- 定时任务、Webhook、跨流程调用。
- 任意代码或脚本节点。
- 自动遍历图片列表。
- 需要人工涂抹或裁剪的无人值守步骤。

后续优先级建议：人工确认节点、条件分支、批量输入映射，再评估并行和循环。

## 10. 验收标准

- 用户可保存并再次打开一条五步示例流程。
- 输入映射只展示类型兼容的变量。
- 流程有错误时不能运行，并可一键定位问题步骤。
- 切换到其他工作区不影响正在运行的流程。
- 页面刷新后可恢复运行状态或查看最终失败原因。
- 失败步骤可重试，上游成功步骤不会重复计费执行。
- 最终图片进入历史生成，并能追溯到流程、运行和步骤。
